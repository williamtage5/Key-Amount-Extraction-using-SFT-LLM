### ⚠️ 优化建议（两个关键点）

虽然可以直接用，但如果你想让模型更完美，请关注以下两点：

#### 1. 关于“共同负担”的金额逻辑 (针对 Sample 3)

**问题现象**：
在 Sample 3 中，总费用是 `11863`，由三被告共同负担。
你的输出是：

* A: 11863 (mutual)
* B: 11863 (mutual)
* C: 11863 (mutual)

**风险点**：
如果你后续做统计，直接 `sum(burden_amount)` 会得到 `11863 * 3 = 35589` 元，导致数据膨胀。
微调模型可能会困惑：*“为什么明明只收了 1万，却记了 3万的账？”*

**优化策略（二选一）**：

* **策略 A（推荐，保留法律准确性）**：**保持现状**。因为在法律上，“共同负担”往往意味着债权人可以找其中任何一人要这笔钱（连带），或者暂未明确分摊比例。只要 `liability_type` 是 `mutual` 或 `joint`，在后续统计代码中做特殊处理（去重计算）即可。
* **策略 B（便于统计，但有风险）**：**强行均摊**。让 LLM 输出时直接除以 3（即每人 3954.33）。**我不建议这样做**，因为 LLM 做除法容易错，且法律上不一定是平均分。

**结论**：保持现状即可，但在使用数据时要记住：**`mutual` 类型的金额不能直接累加**。

#### 2. 全角字符与标点清洗 (针对 Sample 4)

**问题现象**：
在 Sample 4 中，金额出现了 `906．5`。
注意这个小数点 `．` 是**全角字符（中文标点）**，而不是半角的 `.`。

**风险点**：
Python 的 `float("906．5")` 在某些版本或库中会报错，或者微调模型会混淆全角和半角数字，导致生成的 JSON 不合法。

**优化代码建议**：
建议在 `main_runner.py` 写入结果前，或者在 `extraction_service.py` 拿到 LLM 结果后，加一个极小的清洗步骤：

```python
def normalize_amount(amount_val):
    """将全角数字/标点转为半角，确保是合法的 float"""
    if isinstance(amount_val, str):
        # 替换全角点
        amount_val = amount_val.replace("．", ".")
        # 还可以处理其他全角数字（如果 DeepSeek 偶尔输出全角）
    return amount_val

```
