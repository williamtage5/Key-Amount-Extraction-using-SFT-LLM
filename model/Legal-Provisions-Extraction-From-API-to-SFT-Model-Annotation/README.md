# 文本数据批量标注与模型蒸馏标注
本阶段实验主要解决原先在法律条款标注与预测的数据集构建的的问题。
之前暂未解决的法律条款预测问题示意在图中：
![alt text](image.png)
本次优化主要解决了：
* 解决了原先使用正则表达式抽取法律和条款时，出现的不符合常规的标签。例如“或者能够证明属于《中华人民共和国婚姻法》”
* 解决法律条款的命名的统一性。例如“最高人民法院《关于人民法院审理借贷案件的若干意见》”和“《最高人民法院关于人民法院审理借贷案件的若干意见》”

有关多分类模型部分我使用LightXML模型，目前还处在辛苦的代码修改和训练当中。

## 实验步骤

整体采用之前的完整的借助外部API调用AI生成数据集，然后使用LLaMA微调Qwen模型，得到一个轻量的标注模型，实现给出一段文本，返回结构化的标签信息。

相较于之前的流程，本次实验更加的规范化，主要体现在：
* 调用之前的训练好的部分划分模型。使得只有“理由区”的文本才被作为输入。精准定位减少数据输入，从而降低训练难度。
* 增加了提示词测试部分，在代码中反复调用得到最优提示词。人为判断通过后才会进入批量标注。
* 仍然使用ollama辅助本地调用模型。在本地实现了将huggingface格式的大模型（15G）整理成ggif 4-bit（4.2G）,从而能够在消费级显卡上成功调用模型（每个样本推理平均花费3.8s）
  
整体的实现的时间的花销在于：
* 在免费的AI模型下找到最优的提示词
* 批量推理得到3w+的样本需要API相应时间
* 我已经整理好之前的子项目，例如7个功能分区划分、调用硅基流动API构造数据集、SFT数据集构造等，一共8个子项目上传到了**私人**的github，并且每个项目的写好了README，方便后续整理和调用。

* LLaMA训练很耗费时间很吃服务器的环境，我遇到并解决了有关检查点保存、量化等级无法变更导致粗精度模型在服务器上无法导出等问题。在本地，遇到了如何转换粗精度模式的问题，以及认识到在消费级显卡上调用训练好的模型的最优解决方案是ollama，因为其配置可以很好的节约显存。

整体的实验步骤如下：
0. 明确目标是想要得到一个模型，给出一整篇裁判文书，能够自动提取出其判决参考的法律条文。是一个识别标注问题。
   1. 难点1：我将如何输入文本，文本截断难度大，有可能截取了太多的无效信息，也有可能由于文本太长没有截取到含有法律条文的片段。这就体现出分区模型的作用。
   2. 难点2：每个判决书的参考的法律条文情况过多（目前3w样本的“法律+条款”组合就大540+类），并且通常一个裁判文书参考多个法律条款。之前识别费用句的BERTQA模型以及识别当事人和委托人的NER模型完全失效，要么没办法得到结构化的返回，要么只提取出部分的信息（并且出现法律和条款不对应的情况。）
1. 调用分区模型，运行parts segment部分，提取出“判决理由区”
2. 以“判决理由区”的文本，运行create train dataset的create train dataset\test_runner.py部分，找到最优的模型以及最优的提示词。目前的模型依然采用的是硅基流动免费的deepseek和qwen的融合模型。
3. 确定好提示词后，设置多个api key，多进程请求返回，整理并得到每个样本json格式的输出。
4. 根据每个案件的结构化标记连接对应提取出的“判决理由区”，进行筛选，构造SFT数据集。
5. 租用AutoDL服务器，下载LLaMA-Factory-v5的docker，将数据集上传到服务器，先再次调整数据集到json格式，然后下载Qwen-7B-Instruct模型作为基底，epoch选择为6，提示模板选择 qwen，学习率设置为 1e-4，LoRA 秩设置为16，LoRA 缩放设置为32。然后下载完整模型到本地，在本地整理成4-bit的gguf格式，分配给ollama。训练曲线为：![alt text](0986d36effe39e59931225686df6fbe6.png)。部分标注示例为：![alt text](b9b51968c33a66988bb0cce68ae27691.png)
6. 在本地调用使用use SFT model to annotate调用配置好的微调模型。整个过程是输入一篇完整的文书，它会先经过分区提取出判决理由区”，输入给ollama，输出json，然后拼接成完整的特征标签数据，最终输出data\SFT model annotation在形成数据集。最终的数据格式是：![alt text](image-3.png)
